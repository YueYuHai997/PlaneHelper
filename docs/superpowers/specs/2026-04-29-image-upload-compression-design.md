# 图片上传自动压缩设计

## 背景

当前项目的图片上传链路为：

1. 前端 `frontend/src/composables/useGardenData.js` 通过 `FormData` 将用户选择的图片直接上传到 `POST /api/upload`
2. 服务端 `server/src/app.js` 使用 `multer.diskStorage` 将原始文件直接写入 `storage/uploads`
3. 后续植物封面、养护日志、AI 图像分析都复用该落盘图片路径

现状问题：

- 上传原图直接落盘，磁盘占用增长快
- AI 分析读取的是原图，图片尺寸过大时会增加图片输入体积，放大 Token 消耗
- 上传链路没有统一的图片压缩收口点，后续难以治理

## 目标

本次优化目标：

1. 所有新上传的照片自动压缩后再落盘
2. 在不明显影响使用体验的前提下，显著降低图片体积
3. 不修改现有前端上传调用方式和接口响应结构
4. 让后续 AI 分析默认使用压缩后的图片，间接降低 Token 消耗

非目标：

- 不迁移或重压缩历史已上传图片
- 不引入前端压缩逻辑
- 不修改植物、日志、AI 业务接口结构

## 方案选型

### 方案 A：仅服务端统一压缩落盘

做法：

- 上传接口改为先接收文件缓冲区
- 服务端统一执行缩放、格式转换和质量压缩
- 压缩后再写入 `storage/uploads`

优点：

- 所有上传入口统一生效
- 压缩结果稳定，不依赖浏览器能力
- 后续 AI 使用的就是压缩文件，节省磁盘和 Token

缺点：

- 服务端上传时会增加少量 CPU 开销
- 需要新增图片处理依赖

### 方案 B：仅前端压缩后上传

优点：

- 上传流量会下降
- 服务端无需处理图片

缺点：

- 不同浏览器兼容性和压缩质量控制更麻烦
- 如果未来新增上传入口，容易漏掉
- 服务端没有统一兜底，不适合作为治理边界

### 方案 C：前后端双重压缩

优点：

- 最稳，网络和存储都能优化

缺点：

- 当前项目体量下收益不成比例
- 复杂度偏高，属于过度设计

### 结论

采用方案 A。

理由：

- 这是最小变更且收益最大的收口点
- 可以在不改前端调用方式的前提下解决核心问题
- 后续如果要补前端压缩，也可以在此基础上渐进增强

## 详细设计

## 服务端上传链路调整

当前 `multer.diskStorage` 会先把原始文件写盘，这不利于压缩前治理。改为：

1. `multer` 使用内存存储，先拿到 `request.file.buffer`
2. 上传路由在收到文件后，调用图片压缩服务
3. 压缩服务输出目标文件名、目标格式和压缩后缓冲区
4. 服务端将压缩后的内容写入 `storage/uploads`
5. 路由继续返回现有格式：
   - `path`
   - `originalName`

这样前端和其他业务调用方无需改动。

## 压缩规则

首版使用固定规则，避免参数过多：

- 目标格式：`jpeg`
- 最长边限制：`1600px`
- 质量：`78`
- 自动旋转：根据 EXIF 修正朝向
- 去除元数据：不保留原始 EXIF 等非必要信息

设计理由：

- `jpeg` 兼容性最好，浏览器和 AI 模型输入都稳定
- `1600px` 足以满足植物识别、病斑观察和日常展示，没必要保留手机原图 3000px 以上尺寸
- `78` 属于体积和清晰度之间较稳的折中值

说明：

- 若源图存在透明通道，统一转 `jpeg` 会丢失透明背景；但当前业务是实拍植物照片，不是设计素材，这个取舍可接受
- 若后续出现透明素材需求，再扩展为“有透明通道输出 png/webp，无透明通道输出 jpeg”

## 文件命名与路径

继续沿用现有随机命名策略，但扩展名固定改为 `.jpg`：

- 命名格式：`时间戳-随机数.jpg`
- 响应路径格式：`/uploads/<filename>.jpg`

原因：

- 路径风格保持不变
- 有利于后续统一识别和清理

## 异常处理

需要覆盖以下场景：

1. 未上传文件
   - 返回 `400`
   - 消息保持清晰，例如 `Image file is required`

2. 上传的不是可解析图片
   - 返回 `400`
   - 消息例如 `Invalid image file`

3. 图片压缩失败或写盘失败
   - 返回 `500`
   - 消息沿用统一错误处理中间件输出

4. 超大文件
   - 首版建议增加 `multer` 文件大小限制，例如 `10MB`
   - 需要显式处理 `MulterError`，超限时返回明确错误，避免恶意或误上传超大原图

5. MIME 类型伪装或损坏图片
   - 不只依赖扩展名和 `accept="image/*"`
   - 以 `sharp` 实际能否解析作为最终判定依据

## 对 AI Token 的影响

当前 AI 分析会读取 `/uploads/...` 对应的本地文件内容并转为图片输入。上传即压缩后：

- 图片像素会下降
- 文件体积会下降
- 模型接收的图像负担会下降

该优化不能精确保证 Token 固定减少比例，因为不同模型对图片计费方式不同，但它会稳定降低高分辨率图片带来的无效开销。

## 依赖与模块划分

新增服务端依赖：

- `sharp`

Windows 安装方式：

```powershell
npm --prefix server install sharp@0.34.5
```

模块划分建议：

1. `server/src/app.js`
   - 调整 `multer` 配置，改为内存存储和大小限制

2. `server/src/routes/upload.js`
   - 负责参数校验、调用压缩服务、写盘、返回响应

3. 新增 `server/src/services/imageCompressionService.js`
   - 封装图片压缩逻辑
   - 负责缩放、格式转换、质量设置和目标文件名生成

说明：

- 新增文件是必要的，因为图片压缩属于独立职责，塞进路由会让上传逻辑变脏
- 这不是无意义拆分，而是保持单一职责

## 测试策略

按回归风险设计以下测试：

1. 上传图片成功后，返回 `201`
2. 压缩后的文件实际写入磁盘
3. 输出文件扩展名为 `.jpg`
4. 非图片输入会被拒绝
5. 未上传文件返回 `400`
6. 超大文件触发限制时返回明确错误

测试分层：

- 单元测试：
  - `imageCompressionService` 的输出格式、缩放规则、异常行为
- 接口测试：
  - `POST /api/upload` 的端到端行为

为了保证测试稳定，测试图片使用程序生成的小尺寸图片缓冲区，不依赖外部文件资源。

## 风险与约束

1. `sharp` 是原生依赖
   - 需要确认当前 Windows Node 环境可正常安装
   - 这是可控风险，`sharp` 在该场景属于成熟方案

2. 压缩会引入有损失真
   - 但目标是植物照片记录，不是印刷素材，风险可接受

3. 首版统一转 `jpeg`
   - 会丢失透明通道
   - 结合当前业务可接受，后续如需求变化再扩展

## 验收标准

满足以下条件即可认为本次设计达标：

1. 新上传图片不再以原始尺寸和原始质量直接落盘
2. 前端上传植物封面和日志照片流程无需改动即可继续使用
3. AI 分析链路继续可用，且默认读取压缩后的图片
4. 自动化测试覆盖上传成功和异常场景
5. 不影响既有数据结构和页面展示逻辑

## 实施顺序

1. 新增压缩服务和测试
2. 调整上传中间件为内存存储
3. 重写上传路由，将“接收文件”改为“压缩后写盘”
4. 补充错误处理分支
5. 运行服务端测试与前端基础构建验证
