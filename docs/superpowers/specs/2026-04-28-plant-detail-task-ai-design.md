# 植物详情视图、自动任务调度与 AI 建档设计

## 目标

在保持当前 `Vue 3 + Express + JSON` 架构不变的前提下，完成以下能力收敛：

1. 植物点击后不再弹窗展示详情，而是在 `PlantsPage` 内切换为全局详情视图。
2. `ManagePage` 只保留任务、日历与季节提示，原“生长日志”迁移到植物详情页中的“生长记录”模块。
3. 养护任务改为以后端规则自动生成与重算，手动任务与自动任务不再并存冲突。
4. 新增植物时必须先通过 Doubao 生成种植建议、常见问题和养护计划，生成成功后才允许落库。
5. AI 生成内容在植物详情页内可人工编辑。

## 范围

本次设计包含：

- 前端页面结构重构
- 后端植物、任务、日志与 AI 建档接口调整
- 本地 JSON 数据结构扩展
- 自动任务调度规则

本次设计不包含：

- 图片问答
- 现有 AI 聊天链路重构
- 多用户、数据库、权限系统
- 删除图片文件的垃圾回收策略

## 推荐方案

采用“渐进式重构，保留 `plants/manage` 两个一级页面”的方案。

原因：

- 改动范围集中，能最大程度复用现有数据加载与布局代码。
- 详情展示改成 `PlantsPage` 内部的双态切换即可，不必再引入新的一级路由。
- 任务自动生成和日志补写全部收口到后端，避免前端和后端各自算一套。

不采用单页面大工作台方案，因为它会把植物、任务、日志、日历和 AI 内容全部塞进一个视图，状态复杂度过高。

## 页面结构设计

### 1. PlantsPage

`PlantsPage` 从“列表 + 详情弹窗”改为“两态页面”：

- 列表态：显示植物卡片列表和“添加植物”按钮。
- 详情态：替换主内容区，显示单株植物的完整详情视图。

详情态内不再使用模态弹窗，而是在当前页面中展示以下模块：

1. 顶部信息区
   - 返回植物列表按钮
   - 植物主图
   - 名称、学名、状态、种植日期、标签
   - 主要操作按钮

2. 主内容区
   - `种植建议`
   - `常见问题`
   - `养护计划`
   - `生长记录`

3. 侧边信息区
   - 当前周期设置
   - 下次浇水/施肥/修剪/换盆日期
   - 状态与备注编辑入口

布局建议采用 `2:1` 双列结构，主列展示 AI 内容与生长记录，侧列展示状态与周期控制，避免内容过度拥挤。

### 2. ManagePage

`ManagePage` 删除“生长日志”相关 UI，只保留：

- 今日任务
- 近期任务
- 日历规划
- 季节提示

其职责仅限于“查看和完成任务”。用户完成任务后，后端自动补写生长记录，不再在该页面手工新增日志。

### 3. 添加植物弹窗

“添加植物”入口可以继续保留模态弹窗。

但点击“添加植物”提交后，流程改为：

1. 上传图片
2. 后端调用 Doubao 生成 AI 建档内容
3. 生成成功后写入植物与首批任务
4. 前端刷新列表并保留在植物页

若 AI 生成失败，则整个新增失败，不写入任何植物数据。

## 数据结构设计

### plants.json

植物对象扩展为：

```json
{
  "id": 1714280000000,
  "name": "圣女果",
  "species": "Solanum lycopersicum",
  "photo": "/uploads/plant-1714280000000.jpg",
  "date": "2026-04-28",
  "status": "healthy",
  "note": "阳台种植",
  "tags": ["蔬果", "喜阳"],
  "schedules": {
    "water": 5,
    "fertilize": 14,
    "prune": 30,
    "repot": 365
  },
  "carePlan": {
    "lastWateredAt": "2026-04-28",
    "lastFertilizedAt": "",
    "lastPrunedAt": "",
    "lastRepottedAt": ""
  },
  "aiProfile": {
    "advice": ["建议 1", "建议 2"],
    "faq": [
      {
        "q": "问题 1",
        "a": "回答 1"
      }
    ],
    "plan": {
      "summary": "整体养护摘要",
      "water": "浇水建议",
      "fertilize": "施肥建议",
      "prune": "修剪建议",
      "repot": "换盆建议"
    },
    "generatedAt": "2026-04-28T10:00:00.000Z"
  }
}
```

说明：

- `schedules` 负责定义周期。
- `carePlan.lastXxxAt` 是计算下一次任务的锚点。
- `aiProfile` 负责承载 AI 生成且允许人工编辑的资料。

### tasks.json

任务对象扩展为：

```json
{
  "id": 1714280000001,
  "type": "water",
  "plant": 1714280000000,
  "date": "2026-05-03",
  "done": false,
  "note": "",
  "source": "auto"
}
```

规则：

- 同一植物、同一任务类型，只允许存在一条未完成任务。
- `source` 可取 `auto` 或 `manual`，但都代表该类型的“下一次待办”。

### logs.json

日志对象扩展为：

```json
{
  "id": 1714280000002,
  "plant": 1714280000000,
  "date": "2026-04-28",
  "text": "已完成浇水，系统已同步下一次浇水任务。",
  "imgs": [],
  "source": "system"
}
```

规则：

- 用户手工观察记录：`source = "user"`
- 任务完成后自动补写记录：`source = "system"`

前端统一展示在“生长记录”时间线中，不强制分列表，但可按 `source` 做轻量样式区分。

## 自动任务调度规则

### 1. 新增植物

植物保存成功后，系统按照 `schedules` 为每种任务类型生成一条未完成任务：

- `water`
- `fertilize`
- `prune`
- `repot`

基准时间优先取植物的 `date`。

例如：

- 植物种植日期：`2026-04-28`
- 浇水周期：`5`
- 首次自动浇水任务：`2026-05-03`

### 2. 完成任务

用户完成某个任务时：

1. 将该任务标记为完成
2. 根据任务类型更新植物的 `carePlan.lastXxxAt`
3. 写入一条 `source = "system"` 的生长记录
4. 删除同植物同类型原有未完成任务
5. 按“实际完成日期 + 当前周期”生成新的下一次任务

### 3. 提前完成 / 延迟完成

始终以“实际完成日期”为准，而不是原计划日期。

例如浇水周期为 5 天：

- 原计划 `2026-05-03`
- 若实际 `2026-05-01` 完成，则下一次改为 `2026-05-06`
- 若实际 `2026-05-06` 完成，则下一次改为 `2026-05-11`

### 4. 手动新增任务

因为用户已明确选择“同类型只保留一条下一次待办”，所以手动新增任务时：

1. 删除同植物同类型现有未完成任务
2. 写入新的手动任务

手动任务不会与自动任务并存。

### 5. 修改周期

修改周期后，系统应立即重算该类型任务的下一次日期。

计算规则：

- 优先使用 `carePlan.lastXxxAt`
- 若该值为空，则回退到植物 `date`
- 若二者都为空，则使用当天日期作为临时基准

更新流程：

1. 更新植物 `schedules`
2. 删除同植物同类型未完成任务
3. 按新周期重新生成下一次任务

## AI 建档设计

### 目标

新增植物时通过 Doubao 一次性生成以下字段：

- `aiProfile.advice`
- `aiProfile.faq`
- `aiProfile.plan`

### 请求时机

在 `POST /api/plants` 内部同步调用。

只有 AI 建档成功，植物才允许写入 `plants.json`。

### 输入上下文

传给模型的信息包括：

- 植物名
- 品种/学名
- 种植日期
- 当前状态
- 标签
- 用户备注
- 默认养护周期

### 输出格式

模型必须返回结构化 JSON：

```json
{
  "advice": [
    "建议 1",
    "建议 2"
  ],
  "faq": [
    {
      "q": "常见问题 1",
      "a": "回答 1"
    }
  ],
  "plan": {
    "summary": "整体养护摘要",
    "water": "浇水建议",
    "fertilize": "施肥建议",
    "prune": "修剪建议",
    "repot": "换盆建议"
  }
}
```

### 校验规则

后端必须校验：

- `advice` 为非空字符串数组
- `faq` 为数组，且每项包含 `q` 与 `a`
- `plan` 为对象，且至少包含 `summary / water / fertilize / prune / repot`

若结构非法，则视为生成失败，不写入植物。

### 编辑规则

AI 内容保存后允许人工编辑。

后端更新植物时，允许前端提交更新后的：

- `aiProfile.advice`
- `aiProfile.faq`
- `aiProfile.plan`

AI 只负责首次生成，不覆盖用户后续手工修改。

## 接口改造设计

### 1. POST /api/plants

新增行为：

- 校验基础字段
- 调用 Doubao 生成 AI 建档内容
- 初始化 `carePlan`
- 生成首批自动任务
- 成功后同时写入植物和任务

失败行为：

- AI 生成失败时返回错误
- 不写入植物
- 不写入任务

### 2. PUT /api/plants/:id

新增行为：

- 支持更新 `aiProfile`
- 支持更新 `schedules`
- 若 `schedules` 变化，则立即重算对应未完成任务

### 3. POST /api/tasks

新增行为：

- 新建任务前，删除同植物同类型未完成任务
- 写入新任务时标记 `source`

### 4. PATCH /api/tasks/:id/toggle

新增行为：

- 从简单“切 done”升级为“任务完成编排入口”
- 若从未完成切到完成：
  - 更新植物 `carePlan`
  - 写系统日志
  - 生成下一次任务
- 若需要支持反向撤销完成，则本期不做，避免规则复杂化

### 5. POST /api/logs

继续保留用户手工记录能力，但入口从 `ManagePage` 挪到植物详情页内。

### 6. 新增 AI 生成服务方法

建议在现有 AI service 中增加独立方法，例如：

- `generatePlantProfile(payload)`

不要把“植物资料建档”和“AI 园丁聊天”混成一个接口语义。

## 前端改造设计

### 1. PlantsPage 拆分

建议拆成以下组件：

- 植物列表组件
- 植物详情页面组件
- 植物概览头部组件
- AI 建议模块组件
- 常见问题模块组件
- 养护计划模块组件
- 生长记录模块组件

原因：

- 当前 `PlantsPage.vue` 已明显过重。
- 若继续堆叠，会把详情视图、编辑逻辑、周期控制、AI 内容、日志时间线全部塞进一个文件里，维护成本过高。

### 2. ManagePage 收缩

删除：

- 日志筛选
- 日志新增表单
- 日志时间线

保留：

- 今日任务列表
- 近期任务列表
- 日历
- 季节提示

### 3. useGardenData

需要扩展：

- 植物详情态所需的方法
- 植物新增后 AI 建档结果接收
- 周期修改后的刷新
- 任务完成后的刷新

## 错误处理

必须显式处理以下场景：

1. AI 建档超时或返回结构非法
   - 前端提示新增失败
   - 不保存植物

2. 用户未填写植物名
   - 前端拦截
   - 后端继续兜底校验

3. 周期为空、0 或非数字
   - 后端归一化并拒绝非法值

4. 找不到植物或任务
   - 后端返回 404

5. 任务完成时植物被删除
   - 后端返回错误，不再写日志或新任务

## 测试策略

### 后端

需要补充测试覆盖：

- 新增植物时 AI 成功，植物和首批任务同时写入
- 新增植物时 AI 失败，不写任何数据
- 完成浇水任务后：
  - `lastWateredAt` 更新
  - 生成系统日志
  - 下一次浇水任务重算
- 提前/延迟完成任务时，下一次日期正确
- 修改周期后，下一次任务正确重算
- 手动新增同类型任务时，旧未完成任务被替换

### 前端

需要验证：

- 点击植物后切换到全局详情视图
- 从详情页返回后恢复列表态
- `ManagePage` 不再显示日志模块
- 生长记录中能显示系统自动日志
- 添加植物时等待 AI 建档结果
- AI 建档失败时不把半成品植物显示到列表中

## 验收标准

满足以下条件即视为完成：

1. 点击植物后进入 `PlantsPage` 内的全局详情视图，不再弹窗。
2. `ManagePage` 中不再存在生长日志录入与时间线。
3. 生长记录统一出现在植物详情页中。
4. 完成浇水任务后，会自动新增一条系统日志到该植物生长记录。
5. 浇水任务会随实际完成日期和周期变化自动重算。
6. 手动新增同类型任务会替换旧的未完成任务。
7. 修改周期后，下一次任务日期会同步更新。
8. 新增植物时必须先完成 AI 建档，成功后才保存。
9. 植物详情中的 AI 建档内容允许人工编辑。
